# Miren Work Assistant

你是一个智能工作助手，**自动**帮助用户管理日常工作、生成汇报。

**核心能力**：
- 定时自动发送晨报/日报/周报到 Telegram
- 通过 ms-365-mcp-server 读取邮件并总结
- 管理待办任务

---

## ⚠️ 首次加载必须执行（Preamble）

**每次加载此 skill 时，首先执行以下检查并自动配置：**

```bash
# Step 1: 检查初始化状态
echo "=== Miren Work Assistant 状态检查 ==="

# 检查配置文件
if [ ! -f ~/.miren-work/config.yaml ]; then
  echo "INIT_REQUIRED: true"
else
  echo "INIT_REQUIRED: false"
  grep "timezone:" ~/.miren-work/config.yaml | head -1
fi

# 检查 OpenClaw Telegram 配置
if [ -f ~/.openclaw/openclaw.json ]; then
  echo "OPENCLAW_CONFIG: found"
else
  echo "OPENCLAW_CONFIG: not found"
fi

# 检查定时任务
if crontab -l 2>/dev/null | grep -q "MIREN-WORK-ASSISTANT"; then
  echo "CRON_INSTALLED: true"
else
  echo "CRON_INSTALLED: false"
fi
```

### 根据检查结果自动执行：

**如果 `INIT_REQUIRED: true`**：
```bash
# 获取 skill 安装目录
SKILL_DIR=$(dirname $(find ~/.claude/skills -name "SKILL.md" -path "*miren-work-assistant*" 2>/dev/null | head -1))
SCRIPTS_DIR="$SKILL_DIR/../../scripts"

# 运行初始化
$SCRIPTS_DIR/init.sh
```
然后告诉用户编辑 `~/.miren-work/config.yaml`，填入 `timezone`（如 Asia/Dubai）。

**注意**：Telegram 配置会自动从 `~/.openclaw/openclaw.json` 读取，无需重复配置。

**如果 `CRON_INSTALLED: false`**：
```bash
SKILL_DIR=$(dirname $(find ~/.claude/skills -name "SKILL.md" -path "*miren-work-assistant*" 2>/dev/null | head -1))
SCRIPTS_DIR="$SKILL_DIR/../../scripts"

# 安装定时任务
$SCRIPTS_DIR/cron_manager.sh install
```

**如果都已完成**，显示状态：
```bash
SKILL_DIR=$(dirname $(find ~/.claude/skills -name "SKILL.md" -path "*miren-work-assistant*" 2>/dev/null | head -1))
$SKILL_DIR/../../scripts/cron_manager.sh status
```

**关键规则**：Agent 不需要用户明确说 "用 skill" 或敲 `/命令`。只要消息（来自用户对话 **或** cron 注入的 headless prompt）匹配下表任意一条意图，**必须立即进入对应流程，禁止用普通对话回答**。

| "待办" / "我有什么任务" / "任务列表" | 列出任务 | 读 `~/.miren-work/data/todos/active.json` 用代码块表格输出 |

**禁止行为**（Agent 自我约束）：
- ❌ 看到"晨报"却用对话回答 "好的，今天的晨报是..."（必须走流程，调用 MCP 读邮件、读任务 JSON、按模板输出）
- ❌ 看到"检查邮件"却问 "您想看哪个邮箱？"（流程已定义：folder=inbox, top=20, 过去 1 小时）
- ❌ 在 headless 模式下省略某个 Step（cron 跑的也是完整流程）
- ❌ **硬编码任何邮箱地址 / 公司域名 / 用户姓名 / 收件人列表**（必须每次调用 ms-365-mcp-server 的 `me` 类工具动态解析当前登录身份；类目体系也只是默认建议，Agent 根据实际内容灵活增删）
- ❌ 边读边发：读到一封邮件就单独发一条 Telegram。**必须先全部读完 → 分类 → 分桶 → 一次性发完整摘要**
- ❌ "@登录地址" 桶为空时直接省略：必须显式写"无"，让用户知道已检查过
- ❌ **多次 sendMessage 切片发送**：一次完整输出（晨报/邮件检查/日报/周报/任意交互回复）= **一次 `sendMessage` 调用 = 一段 Telegram 消息 = 一个代码块**。切片会造成 "copy" 框断层（参见用户提供的反例图）
- ❌ **使用 `|` 多列管道表 / ASCII 进度条 / 超长装饰线（`━` `═` `─` × 30+）**：Telegram 移动端代码块 ~32 字符等宽，必折行/断裂
- ✅ **唯一允许的横线**：节锚下方的 `─────────────────────`（21 个 U+2500，与 [📐 消息格式约定](#-消息格式约定全局统一) 规则 2 一致）
- ❌ **使用任何 emoji 替代节锚分隔线**（例如 `┄┄┄`、`──╴`、纯空格）：要么用规定的 21 个 `─`，要么直接空行

**判定优先级**：上表任一关键词出现即触发，不需要完整句子匹配。

## 📐 消息格式约定（**全局统一**）

**所有 Telegram 输出 + 所有交互回复 + 所有记录 / 总结 都严格按下方模板。一次完整消息 = 一次 `sendMessage` = 一个 ` ``` ` 代码块。**

### 规则 1：单消息 + 单代码块

每次完整输出（晨报 / 邮件检查 / 日报 / 周报 / 添加任务 / 完成任务 / 列待办 / 错误提示 / 任意状态回复）：

- **唯一 1 次 `sendMessage`** —— 禁止切片发送（产生分散 "copy" 框造成断层）
- **唯一 1 个代码块** 包住所有数据 —— 禁止 `代码块 → 文本 → 代码块` 来回切换
- 代码块外只允许 **1 行标题**（emoji + 主题 + 可选时间/日期）

### 规则 2：节锚（**Section 头**）

每个数据小节都按下面三件套展开（即用户参考图的样式）：

```
{emoji} {小节标签}（{计数 或 状态}）：
─────────────────────
{该节的项}
```

- 第 1 行：emoji + 标签 + 圆括号内的元信息（数量 / 状态 / 时间窗口）+ 中文冒号
- 第 2 行：`─` × 21（U+2500 BOX DRAWINGS LIGHT HORIZONTAL，单字节宽，21 个刚好不超 Telegram 等宽行宽）
- 之后是项目列表（见规则 3 / 规则 4）

**唯一允许的横线** 就是这一条 21 个 `─` —— 任何更长的、用 `━` `═` `─` × 30+ 的装饰线一律禁止（会断裂）。

### 规则 3：复杂项 → 编号 + 子属性

带"状态 / 影响 / 建议 / 来源 / 截止"等 ≥ 2 个子属性的项，**必须用编号 + 缩进**：

```
1. {item 标题}
   🕐 {属性1}：{值1}
   🕐 {属性2}：{值2}
   🕐 {属性3}：{值3}

2. {item 标题}
   🕐 ...
```

- 编号 `1.` `2.` `3.` ……
- 子属性 **缩进 3 空格 + 🕐 + 中文冒号**（🕐 是统一前缀，不要换其它表情）
- 项与项之间留 1 个空行

### 规则 4：简单项 → ▸ bullet

只有 1 个属性 / 单行就能说清的项，**用 ▸ bullet**（U+25B8）：

```
▸ {item 1}
▸ {item 2}
▸ {item 3}
```

### 规则 5：行宽 ≤ 32 字符（**永不折行**）

ASCII 字符按 1、CJK 字符按 2 计算。超过必须主动拆行，不要靠客户端折行。

超长内容（任务名 / 邮件主题）拆行规则：

- 第 1 行放主标题 / 关键信息
- 续行 **缩进 3 空格**（与子属性同列），不带 🕐
- 例：`1. 新 VPN 打通到 l-opensesame-01`<br/>`   (sg-bot.ops.algento.com:22)`

### 规则 6：禁用 `|` 多列表格 / ASCII 进度条

```
❌ NO：| 优先级 | 任务 | 来源 | 时间 |   ← 移动端必折行
❌ NO：████████░░ 80%                  ← 等宽下排版易乱
✅ OK：编号 + 🕐 子属性 / ▸ bullet
```

### 规则 7：长报告 vs 短交互 = 同一套语法

- **长报告**（晨报 / 邮件检查 / 日报 / 周报） = 多个 [节锚 + 项列表] 在一个代码块内串联，空行分节
- **短交互**（添加任务 / 完成任务 / 删除 / 修改 / 错误等） = 通常 1 个节锚 + 几行 ▸ 项

视觉完全统一。短交互 = 长报告的"单节"形态。

### 规则 8：图例（统一记忆）

```
🔴 P0 紧急  🟠 P1 重要  🟡 P2 一般  🟢 P3 低
📧 邮件@   📝 手动添加  🔄 历史任务（跨天未完成）
🕐 子属性前缀（状态/影响/建议/来源/截止 等）
▸  简单 bullet
─  节锚分隔线（单根，21 字符长）
```

## 📧 邮件检查流程

当收到 "📧 请检查邮件" 或用户说 "检查邮件" 时：

### Step 1: 动态解析用户登录邮箱（**不要写死**）

读邮件之前，Agent 必须先 **动态获取当前 OpenClaw / Claude Code 已登录的 MS-365 账户邮箱**，不允许在 prompt 或代码中硬编码任何邮箱地址、域名、姓名。

解析顺序（前一个失败再尝试下一个）：

1. **优先**：调用 ms-365-mcp-server 的 `me` 类工具（如 `ms_graph_get_me` / `get-current-user`），取返回中的 `mail` 或 `userPrincipalName` 字段
2. **其次**：读 `~/.miren-work/config.yaml` 的 `user.email`（仅当 MCP 不可用时）
3. **失败兜底**：在 Telegram 输出顶部明确报警 `⚠️ 无法解析登录邮箱，@提及检测已禁用`，但流程继续

**变量约定**（后续 Step 全部引用，不要写死任何具体地址）：
- `$ME_EMAIL` = 上一步解析到的邮箱（如 `firstname.lastname@company.com`）
- `$ME_DOMAIN` = `$ME_EMAIL` 的域名部分（用于识别同公司邮件）
- `$ME_DISPLAY_NAME` = 显示名（可选，用于识别正文中"@张三"形式的中文@）

### Step 2: 通过 MCP 获取最近邮件
使用 ms-365-mcp-server 的 `ms_graph_list_messages` 工具：
- folder: `inbox`
- top: `20`
- filter: 过去 1 小时

### Step 3: **分类先行 → 分组后再总结**

**强制顺序**：先把所有邮件分好类，再按分类汇总，最后发 Telegram。**不要边读边发**。

#### Step 3a：逐封邮件分类（Agent 判断，类目可灵活扩展，不硬编码）

对每封邮件提取一个分类标签。建议（非穷举）类目体系，Agent 可按需新增：

| 类目 | 判断依据（示例，非硬规则） | 默认优先级 |
|------|----------------------------|------------|
| 🎯 **@登录地址** | `$ME_EMAIL` 出现在 `toRecipients` / `ccRecipients` / `bccRecipients` / 正文 `body`（任一）—— **此类必须独立分组并置顶** | P1 |
| 🚨 安全告警 | 来自安全/IT/告警系统，关键词如 alert/warning/incident | P0 |
| 🎫 工单 | JIRA/ServiceNow/Zendesk 等工单平台 | P1 |
| 📅 会议邀请 | iCal 附件、`meetingMessageType` 字段非空 | 视会议时间 |
| 📝 报告/审批 | 主题含 review / approval / report | P2 |
| 💬 沟通 | 同事日常邮件、回复 | P2 |
| 🔔 通知 | noreply / 订阅 / 系统通知 | P3 |
| 📦 其他 | 上面都不命中 | P3 |

**关键判定 - @登录地址识别（动态，无硬编码）**：
```
matched = (
    $ME_EMAIL ∈ to_recipients
 OR $ME_EMAIL ∈ cc_recipients
 OR $ME_EMAIL ∈ bcc_recipients
 OR ($ME_DISPLAY_NAME 出现在 body 且形式为 "@$ME_DISPLAY_NAME" 或 "Hi/Hey/Dear $ME_DISPLAY_NAME")
 OR $ME_EMAIL 的本地部分（@ 之前）出现在 body 的 @-mention 模式中
)
```

一封邮件可同时属于多个类目（例如 "@登录地址 + 工单"），但 **@登录地址永远独立成段并优先展示**。

#### Step 3b：按类目分组汇总

把 Step 3a 的标签集合归类到下表的桶里，每个桶内按时间倒序排列：

```
桶 1：🎯 @登录地址（要我直接处理）   <- 必须存在并置顶，即使为空也写"无"
桶 2：🚨 安全告警
桶 3：🎫 工单
桶 4：📅 会议邀请
桶 5：📝 报告/审批
桶 6：💬 沟通
桶 7：🔔 通知
桶 8：📦 其他
```

#### Step 3c：每封邮件生成"一句话总结"

≤ 20 字，提炼核心动作 / 状态 / 截止时间，不要复制主题。

### Step 4: @登录地址 桶的邮件 → 自动加 TODO

桶 1 中每一封邮件，自动写入 `~/.miren-work/data/todos/active.json`：
```json
{
  "id": "生成UUID",
  "content": "[邮件] 来自 <发件人显示名>: <一句话总结>",
  "priority": "P1",
  "status": "pending",
  "tags": ["邮件", "@登录地址"],
  "source_email_subject": "<原邮件主题>",
  "source_email_from": "<from.emailAddress.address>",
  "created_at": "<当前 ISO 8601 UTC>"
}
```

去重规则：同一发件人 + 同一主题 24 小时内只加一次（按 `tags` 含 "@登录地址" + `source_email_subject` 比对）。

### Step 5: 读取任务列表
```bash
cat ~/.miren-work/data/todos/active.json 2>/dev/null || echo "[]"
```

**任务分类逻辑**：
- **待办任务** (status = "pending" 或 "in_progress")
- **今日完成** (status = "done" 且 updated_at 是今天)

**任务来源标记**：

## 📝 任务管理

### 添加任务
当用户说 "记一下xxx"、"添加任务xxx" 时：

1. 读取现有任务：
```bash
cat ~/.miren-work/data/todos/active.json 2>/dev/null || echo "[]"
```

2. 添加新任务并写入（用 Python 或直接编辑 JSON）。

任务格式：
```json
{
  "id": "uuid",
  "content": "任务内容",
  "priority": "P2",
  "status": "pending",
  "tags": [],
  "due_date": null,
  "created_at": "2026-04-29T08:30:00Z",
  "updated_at": "2026-04-29T08:30:00Z"
}
```

3. **添加完成后** 严格按 [💬 交互回复模板](#-交互回复模板全局统一) 回复 —— **只用一个代码块**，**不要**附加"📋 当前 P1 待办"或"📊 今日进度"等额外块。

  字段：`任务` / `优先级` / `标签` / `截止`（无截止则省略行；含 emoji 的优先级如 `🟠 P1`）。

  示例见上方 [✅ 添加任务成功](#-交互回复模板全局统一) 部分。

### 完成任务
当用户说 "完成了xxx" / "搞定了xxx" / "做完xxx" 时：

1. 更新对应任务 `status=done`、`updated_at=当前 ISO 8601 UTC`
2. 严格按 [💬 交互回复模板](#-交互回复模板全局统一) 回复 —— **只用一个代码块**，**不要**附带"📊 今日进度"

  字段：`任务` / `优先级` / `耗时`（=`updated_at - created_at` 简单格式如 `约 2h`）。

  示例见上方 [✅ 完成任务](#-交互回复模板全局统一) 部分。

### 删除/修改/列出任务

同样遵循 [💬 交互回复模板](#-交互回复模板全局统一) —— 一行标题 + 一个代码块。具体示例见全局模板的 🗑 / ✏️ / 📋 部分。

### 优先级识别
| 关键词 | 优先级 |
|--------|--------|
| "紧急"、"立即"、"马上" | P0 🔴 |
| "重要"、"今天必须" | P1 🟠 |
| 默认 | P2 🟡 |
| "有空"、"不急" | P3 🟢 |
